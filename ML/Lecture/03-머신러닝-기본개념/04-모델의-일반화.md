# 4강. 모델의 일반화 — 과적합 · 편향-분산 · 데이터 분할 · 데이터 누수

3강에서 "손실을 줄이는 게 학습"이라고 배웠습니다. 그런데 여기에 함정이 있습니다.
**훈련 데이터의 손실만 죽어라 줄이면, 정작 실전에서 망가지는 모델이 만들어집니다.**

이 강의는 챕터 3 전체를 관통하는 한 문장으로 시작합니다.

> **머신러닝의 진짜 목표는 "훈련 데이터를 잘 맞히는 것"이 아니라,**
> **"처음 보는 데이터를 잘 맞히는 것" = 일반화(generalization)다.**

---

## 1. 과적합 vs 과소적합

### 비유: 시험 공부

- **과소적합(underfitting)** — 공부를 너무 안 해서 기출문제도 못 풉니다.
  모델이 너무 단순해서 데이터의 패턴조차 못 잡은 상태.
- **과적합(overfitting)** — 기출문제의 **답을 통째로 외워버려서**, 숫자만 살짝 바뀐
  새 문제는 못 풉니다. 모델이 훈련 데이터의 **노이즈까지 암기**한 상태.
- **적절(good fit)** — 문제의 **원리**를 이해해서 처음 보는 문제도 풉니다. ✅

### 그래프로 보는 세 가지

```
  과소적합            적절(good fit)         과적합
  (너무 단순)          (원리를 학습)          (노이즈까지 암기)

  y│   • •            y│    ___•            y│    /\  •
   │ •___ •            │  •/   •\•           │ •/  \/\•
   │•/    •            │ •/      \•          │•/ •  •/\•
   └────────→ x        └────────→ x          └────────→ x
   직선 하나로          부드러운 곡선이         모든 점을 억지로
   대충 그음            추세를 잘 잡음          통과하는 구불구불 곡선
```

### 어떻게 구별하나? — 훈련 점수와 검증 점수의 **격차**

| 상태 | 훈련 성능 | 검증(새 데이터) 성능 | 진단 |
|------|:---------:|:--------------------:|------|
| 과소적합 | 낮음 | 낮음 | 모델이 너무 단순 / 학습 부족 |
| **적절** | 높음 | 높음 | 👍 |
| 과적합 | **매우 높음** | **뚝 떨어짐** | 훈련 데이터를 암기함 |

> 💡 **진단의 핵심:** 훈련 성능은 좋은데 검증 성능이 나쁘면 → **과적합**.
> 둘 다 나쁘면 → **과소적합**. 이 한 가지만 봐도 다음 처방이 갈립니다.

### 처방

- **과소적합이면**: 더 복잡한 모델, 특성 추가, 더 오래 학습
- **과적합이면**: 더 많은 데이터, 모델 단순화, **정규화(regularization)**,
  조기 종료(early stopping) — 로드맵 5번에서 Ridge/Lasso로 이어집니다.

## 2. 편향-분산 트레이드오프 (Bias-Variance Tradeoff)

과적합/과소적합을 더 정밀하게 설명하는 이론적 틀입니다. 모델의 오차는 크게
두 성분으로 분해됩니다.

- **편향(Bias)** — 모델이 너무 단순해서 생기는 **체계적 오차**.
  → 높으면 **과소적합**. (계속 같은 방향으로 틀림)
- **분산(Variance)** — 훈련 데이터가 조금만 바뀌어도 모델이 크게 **출렁이는 정도**.
  → 높으면 **과적합**. (데이터에 너무 민감)

### 다트판 비유

```
   낮은 분산           높은 분산
 ┌─────────┐        ┌─────────┐
 │   ⊙     │        │  •   •  │   낮은 편향
 │  (몰림,  │        │ • ⊙  •  │   (중심 근처)
 │  정중앙) │        │   •  •  │
 └─────────┘        └─────────┘
 ┌─────────┐        ┌─────────┐
 │         │        │ •     • │   높은 편향
 │  ●(몰림, │        │•  •     │   (중심에서 벗어남)
 │  치우침) │        │   •   • │
 └─────────┘        └─────────┘
   ⊙ = 정답 중심
```

**트레이드오프인 이유:** 모델을 복잡하게 만들면 편향은 줄지만 분산이 커지고,
단순하게 만들면 분산은 줄지만 편향이 커집니다. 둘을 동시에 0으로 만들 수 없어,
**둘의 합(총 오차)이 최소가 되는 적절한 복잡도**를 찾는 것이 목표입니다.

```
오차│ \                          /  ← 분산 (복잡할수록 증가)
   │  \                       /
   │   \  편향              /
   │    \ (복잡할수록 감소) /
   │     \______      ____/
   │            \____/  ← 총 오차 최소 = 최적 복잡도
   └──────────────────────────→ 모델 복잡도
        단순 ←            → 복잡
```

## 3. 데이터 분할 — 왜 데이터를 나누는가

**"왜 데이터를 다 쓰지 않고 일부러 나누지?"** — 과적합을 *탐지*하려면,
모델이 **학습에 쓰지 않은 데이터**로 채점해봐야 하기 때문입니다.
훈련에 쓴 데이터로 채점하는 건 "기출로 공부하고 같은 기출로 시험 보는 것"과
같아서 진짜 실력을 알 수 없습니다.

### 세 갈래 분할

```
  전체 데이터 100%
 ┌────────────────────────┬──────────┬──────────┐
 │      훈련(Train) 60%    │ 검증 20% │ 테스트 20%│
 └────────────────────────┴──────────┴──────────┘
        ↓                      ↓            ↓
   모델 학습에 사용       모델·하이퍼파라미터    "최종 시험"
   (다이얼 조정)          선택·튜닝에 사용      딱 한 번만,
                         (여러 번 봐도 됨)     절대 학습에 안 씀
```

| 세트 | 역할 | 비유 |
|------|------|------|
| **훈련(Train)** | 파라미터를 학습 | 교과서로 공부 |
| **검증(Validation)** | 모델 선택·튜닝 | 모의고사로 실력 점검 |
| **테스트(Test)** | 최종 성능 보고 | 진짜 수능 (한 번만) |

> ⚠️ **테스트 세트는 "금고에 넣어두는" 데이터입니다.** 모델을 고르거나 튜닝하는
> 데 테스트 세트를 보면, 그 순간 테스트 세트가 사실상 "공부한 문제"가 되어
> 성능을 과대평가하게 됩니다. **최종 보고용으로 딱 한 번만** 꺼내세요.

데이터가 적을 땐 **k-fold 교차검증**으로 검증 세트를 돌려쓰는데, 이는 로드맵 6번
("모델 평가와 검증")에서 자세히 다룹니다. 지금은 **"왜 나누는가"** 의 직관이 핵심.

## 4. 데이터 누수 (Data Leakage) — 가장 위험한 함정

> **모델이 예측 시점에 "원래는 알 수 없어야 할 정보"를 학습 과정에서 몰래
> 엿보는 것.** 결과적으로 평가 점수는 비현실적으로 높지만, 실전에 배포하면
> 처참하게 무너집니다.

데이터 누수는 입문자가 가장 자주 빠지고, **가장 발견하기 어려운** 함정입니다.
점수가 "너무 좋게" 나오면 기뻐하기 전에 누수를 의심해야 합니다.

### 대표 유형 ① — 전처리에서 테스트 정보를 미리 봄 (가장 흔함)

스케일링(표준화)을 할 때 평균·표준편차를 **전체 데이터**에서 계산하면,
훈련 단계에서 이미 테스트 데이터의 통계를 엿본 셈이 됩니다.

```
❌ 잘못: 전체 데이터로 scaler.fit() → 그 후 train/test 분할
✅ 올바름: 먼저 분할 → 훈련 데이터로만 scaler.fit()
          → 그 기준을 train과 test에 각각 transform()
```

> 📌 **철칙: "훈련 데이터에서 배운 규칙을 테스트 데이터에 적용한다."**
> 순서가 항상 train → test 여야 합니다. (로드맵 4번 전처리 챕터의 핵심 원칙)

### 대표 유형 ② — 미래 정보가 특성에 섞임

- 이탈 고객 예측에 "**이탈 후** 해지 위약금" 같은, 결과가 나온 뒤에야 알 수 있는
  값을 특성으로 넣음 → 사실상 정답을 보고 맞히는 것.

### 대표 유형 ③ — 타깃이 특성에 변형되어 숨어 있음

- 예측하려는 타깃을 계산에 쓴 파생 특성을 무심코 포함.

### 누수를 막는 습관

- 분할을 **가장 먼저** 한다. 모든 전처리는 분할 **이후** 훈련 데이터 기준으로.
- "이 특성, 예측하는 그 순간에 정말로 알 수 있는 값인가?"를 매번 자문.
- scikit-learn의 **Pipeline**을 쓰면 전처리가 자동으로 훈련 기준으로만 학습됨
  (로드맵 4·6번에서 등장).

## 스스로 점검

1. 훈련 성능은 95%인데 테스트 성능이 60%다. 무슨 문제이고 처방은?
2. 테스트 세트를 모델 선택에 쓰면 안 되는 이유는?
3. 편향이 높은 모델과 분산이 높은 모델은 각각 과적합/과소적합 중 무엇인가?
4. 표준화(StandardScaler)에서 데이터 누수를 피하려면 평균·표준편차를 어떤
   데이터에서 계산해야 하나?

<details>
<summary>정답 보기</summary>

1. **과적합**. 처방: 데이터 추가, 모델 단순화, 정규화, 조기 종료 등.
2. 모델을 고르는 데 테스트 세트를 쓰면 그 데이터가 사실상 "공부한 문제"가 되어
   최종 성능을 과대평가한다. 테스트는 최종 보고용으로 한 번만.
3. 높은 편향 → 과소적합, 높은 분산 → 과적합.
4. **훈련 데이터에서만** 계산한 뒤, 그 기준을 train·test에 각각 적용한다.
